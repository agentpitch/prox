# Recreation specification

This is the canonical text-only specification for rebuilding pitchProx from scratch.

## 1. Product goal

Build a Windows transparent TCP proxy manager with these properties:

- written in Go;
- local WebUI served by the program itself;
- rules based on `Applications`, `Target hosts`, `Target ports`;
- actions `Direct`, `Proxy`, `Chain`, `Block`;
- ordered rule list with first-match-wins;
- portable config saved next to the executable;
- portable SQLite history saved next to the executable;
- live observability for rules, connections, proxy usage and logs;
- Windows tray icon with live proxied traffic mini-graph.

## 2. Launch model

### Desktop mode

- double-clicking `pitchProx.exe` must request elevation if needed;
- desktop mode must run as **one elevated process** containing runtime + WebUI + tray;
- no second tray helper process is required in desktop mode.

### Service mode

- service mode may exist for SCM usage;
- service mode is headless by design;
- no tray is shown from the service.

## 3. Persistence model

### Config file

Location:

```text
<dir of pitchProx.exe>\pitchProx.config.json
```

Behavior:

- pretty JSON;
- save by writing `*.tmp` and replacing atomically;
- config must be normalized before validation and save.

### History database

Location:

```text
<dir of pitchProx.exe>\pitchProx.history.sqlite
```

Stored there:

- log records;
- closed/blocked/error connection history;
- per-second proxied traffic samples;
- per-second rule activity samples.

The goal is to keep historical investigation data out of RAM.

## 4. Rule model

A rule contains:

- `id`
- `name`
- `enabled`
- `applications`
- `target_hosts`
- `target_ports`
- `action`
- `proxy_id`
- `chain_id`
- `notes`

Rules are stored in order. The first enabled matching rule wins.

### 4.1 Applications syntax

Separators:

- `;`
- `,`
- newline

Quotes preserve separators and spaces inside tokens.

Interpretation:

- `Any` or `*` → any application
- digits only → PID match
- token containing `\` or `:` → full-path wildcard match
- otherwise → basename wildcard match

Must support:

```text
fire*.exe
"*.bin"
"C:\Program Files\JetBrains\*"
12345
```

### 4.2 Target hosts syntax

Must support:

- `Any`
- exact host name
- wildcard host name
- exact IP
- IPv4 wildcard like `192.168.1.*`
- CIDR
- range like `10.1.0.0-10.5.255.255`
- `%ComputerName%`

### 4.3 Target ports syntax

Must support:

- `Any`
- single port
- range

### 4.4 Actions

- `Direct` → no proxy
- `Proxy` → exactly one proxy profile
- `Chain` → exactly one chain
- `Block` → deny

Normalization rules:

- `Direct` / `Block` clear `proxy_id` and `chain_id`
- `Proxy` clears `chain_id`
- `Chain` clears `proxy_id`

## 5. Proxy and chain model

### Proxy profile

Fields:

- `id`
- `name`
- `type` (`http` or `socks5`)
- `address`
- optional `username`
- optional `password`
- `enabled`

### Chain

Fields:

- `id`
- `name`
- ordered `proxy_ids`
- `enabled`

A chain is an ordered sequence of proxy IDs. Dialer wrapping follows that same order.

## 6. Interception and routing pipeline

### 6.1 Interception

Use WinDivert selectively, not as a global all-packets relay.

The required design is:

- if every enabled rule resolves to `Direct`, run in observer-only mode and do not start WinDivert or the transparent listener at all;
- otherwise open a lightweight outbound TCP SYN classifier;
- for each new relevant SYN, identify PID/exe and run a preflight rule match using only PID, executable path, target IP and port;
- if the result is definitively `Direct`, allow the connection through unchanged and do not build a userland relay path;
- only when the result is non-direct or hostname-dependent, create a flow-table record and redirect the connection into the local transparent listener;
- open dedicated per-flow packet interception only for those intercepted flows.

Must avoid loops for:

- loopback;
- traffic to the transparent listener itself;
- self-traffic generated by pitchProx where applicable.

### 6.2 Owner lookup design

Do **not** scan the full TCP owner table repeatedly on every SYN in the hot path.

Use an on-demand owner cache:

- cache exact local+remote tuples;
- keep a local-endpoint fallback map;
- keep a PID→exe cache with TTL;
- refresh the cache on demand when a SYN cannot be resolved from the current maps.

The interception hot path should mostly do cheap map lookups and should not require a hot periodic full-table scan.

### 6.3 Transparent listener

For each accepted redirected socket:

- map the connection back to the flow entry;
- sniff HTTP `Host` or TLS `SNI` where possible;
- compile a request with PID, exe path, host, target IP and port;
- run the full rule engine;
- execute `Direct`, `Proxy`, `Chain`, or `Block`.

Important: the transparent listener is not the default path for all traffic anymore. It exists only for intercepted flows.

### 6.4 Relay and accounting

When the connection is not blocked:

- dial upstream directly, through one proxy, or through a chain;
- relay bytes both ways;
- batch traffic counters instead of writing counters for every copied chunk;
- attribute bytes to the matching rule;
- attribute proxy activity only for `Proxy`/`Chain`.

## 7. Observability design

### 7.1 Active connections

Requirements:

- keep only currently open/opening connections in RAM;
- keep closed history in SQLite;
- show grouped rows by PID/process/host/port/rule/action;
- default `Все` tab shows only explicit non-default rule matches;
- `Ещё` shows default/unassigned traffic;
- support filtering with a free-text search field and by clicking rule cells;
- support action tabs;
- support copying PID/path/host/port.

### 7.2 Log

Requirements:

- real-time delivery via SSE;
- newest items visually on top in the UI;
- persist logs in SQLite;
- keep at most 100 latest entries per process plus a small generic tail in the visible log view;
- `warn/error` always captured;
- `info/debug` captured only while WebUI is active or recently active.

### 7.3 Proxy activity

Requirements:

- show proxied traffic only;
- graph incoming and outgoing bytes per second;
- use the same retention window as the rest of observability.

### 7.4 Rule metrics

Each rule card must show for the retention window:

- connection count
- incoming bytes
- outgoing bytes

Clicking these metrics must focus observability on that rule.

## 8. Quiet idle design

The application must have a quiet idle mode.

When WebUI is not open:

- tray must **not** poll the heavy full snapshot;
- tray must use a lightweight traffic-only view or direct provider;
- tray polling must **not** mark the UI active;
- verbose logging must remain off.

## 9. Tray requirements

Desktop mode tray must:

- show live proxied Rx/Tx traffic;
- render a small filled graph with separate Rx/Tx colors;
- open WebUI on double click;
- show `Управление` and `Выйти` on right click;
- stop the whole desktop runtime on `Выйти`.

## 10. API surface

Minimum required endpoints:

- `GET /api/health`
- `GET /api/config`
- `PUT /api/config`
- `GET /api/snapshot`
- `GET /api/tray`
- `GET /api/events`
- `POST /api/proxy-test`
- `POST /api/control/stop`

## 11. UX requirements

- compact top bar, not sticky;
- settings behind a gear button;
- rule cards open a dialog directly on click;
- rule enable toggle is a compact checkbox without a large label;
- move buttons are compact and below that checkbox;
- disabled rules must be visually obvious;
- right column wider than the left;
- process hover in connections shows full executable path.

## 12. Default config

Must ship with at least:

1. `Localhost / This PC`
   - enabled
   - applications `*`
   - target hosts `localhost; 127.0.0.1; ::1; %ComputerName%`
   - ports `Any`
   - action `Direct`

2. `Default`
   - enabled
   - applications `*`
   - target hosts `Any`
   - ports `Any`
   - action `Direct`

## 13. Known intentional limits

- TCP only
- hostname inference primarily from HTTP/TLS
- no authenticated WebUI
- service mode headless
- no UDP/QUIC/HTTP3

A future rebuilding agent should preserve these limits unless intentionally expanding scope.
